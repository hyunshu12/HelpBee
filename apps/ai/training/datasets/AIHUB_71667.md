## 3. 디렉터리 구조

다운로드 후 압축 해제 시 다음 구조를 가짐 (Sample 기준 검증):

```
{root}/
├── 01.원천데이터/                   # 이미지 (.jpg)
│   ├── 성충/
│   │   ├── 성충_정상/{NNN}/*.jpg
│   │   ├── 성충_응애/{NNN}/*.jpg
│   │   └── 성충_날개불구바이러스감염증/{NNN}/*.jpg
│   └── 유충/
│       ├── 유충_정상/{NNN}/*.jpg
│       ├── 유충_응애/{NNN}/*.jpg
│       ├── 유충_부저병/{NNN}/*.jpg
│       └── 유충_석고병/{NNN}/*.jpg
└── 02.라벨링데이터/                 # 라벨 (.json), 미러 구조
    └── ... (위와 동일)
```

**핵심**: 라벨 경로에서 `02.라벨링데이터` → `01.원천데이터` 치환 + `.json` → `.jpg` 로
이미지 경로가 바로 나옴. `aihub_to_yolo.py:_resolve_image_path()` 가 처리.

### 폴더명 vs 실제 라벨

⚠️ 폴더명 ≠ 라벨. `성충_응애` 폴더 안 사진에도 정상벌·부저병·석고병 등 다양한 클래스가
함께 라벨링되어 있음. **폴더명은 "주된 진단명"**, 라벨은 객체별 정확한 클래스.
변환 시 폴더명을 무시하고 `annotations[].category_id` 만 사용.

```jsonc
    "id": "007",                    // ★ 벌통 ID (분할 group key)
    "PCR": "미국부저, 유럽부저, 날개불구",
    "type": "스티로폼",             // 벌통 재질

    "plate_num": "5장"

    "in_temperature": ..., "in_humidity": ..., "in_carbon_monoxide": ...,
    "out_temperature": ..., "out_humidity": ...
  }

```

### 파일명 패턴

`{기기코드}_{?}_{colony}_{datetime}_{seq*4}.jpg`

예: `C_002_007_20230820105708_001_001_001_001.jpg`
- `C` — 추정 기기 코드 (A/B/C 관찰됨; JSON `collection.device` 가 정확한 한국어 명)
- `007` — colony id (JSON `colony.id` 와 일치)
- `20230820105708` — timestamp

→ 파일명만으로도 메타 추출 가능하지만, **JSON 메타가 정확**하므로 그쪽을 사용.

## 5. 클래스 매핑 — 71667 7-class → HelpBee 3-class

v0.1.0 결정 (Q1' = B+):

| 71667 cat_id | 71667 name | HelpBee class | 우리 id |
|---|---|---|---|
| 0 | 유충_정상 | bee_normal | 0 |
| 4 | 성충_정상 | bee_normal | 0 |
| 1 | 유충_응애 | bee_with_varroa | 1 |
| 5 | 성충_응애 | bee_with_varroa | 1 |
| 2 | 유충_석고병 | bee_other_disease | 2 |
| 3 | 유충_부저병 | bee_other_disease | 2 |
| 6 | 성충_날개불구바이러스감염증 | bee_other_disease | 2 |

이 매핑은 `apps/ai/training/data/aihub_to_yolo.py:CLASS_MAPPING` 단일 소스.

### 왜 3-class인가

다른 옵션과 비교:

| 옵션 | 클래스 수 | 데이터 활용 | 결정 |
|---|---|---|---|
| B′ (응애만 + 정상) | 2 | 88% (질병 392 인스턴스 무시) | ❌ 다른 질병 영역이 false positive 학습 위험 |
| **B+ (3-class)** | 3 | 100% | ✅ 채택 — 응애 정확도 자체도 향상 |
| C (7-class 풀) | 7 | 100% | ❌ 응애·날개불구·석고병이 50개대 — 학습 어려움 |

v0.2.0+ 에서 데이터 확장 후 7-class 분리 검토.

## 6. ⚠️ 중요한 발견 — Q3 라벨 케이스 = B

샘플 330장 / 응애 인스턴스 100개 분석 결과:

| 응애 클래스 bbox 통계 | 값 |
|---|---|
| 최소 면적 비율 (이미지 대비) | **0.13** (805×323 px) |
| median | 0.39 |
| 최대 | 0.96 (1911×1037 px) |
| ratio < 0.005 (응애 자체 추정) | **0개** |

→ **응애 자체의 bbox는 한 개도 없다.** 모든 응애 라벨은 "응애가 보이는 벌(또는 프레임 일부)
영역" — 한 마리 단위의 큰 bbox.

### 디자인 영향

1. **클래스 명명**: `varroa_mite`(응애 객체) → `bee_with_varroa`(감염된 벌 영역) 로 정확화.
2. **VMIR 직접 계산 불가**: 응애 자체 카운트가 안 됨. 대신 **infestation_rate**
   = 감염 영역 인스턴스 / 전체 벌 인스턴스. 임계값(3%/10%)은 차용하되 베타에서 보정 필요.
3. **P2 head 가치 재검토**: 모든 bbox 가 medium~large 라 P2(작은 객체용)는 v0.1.0 데이터에서
   필수 아님. 다만 실제 양봉가 사진은 더 멀리서 찍어 작게 나올 수 있음 → 베타 비교 후 결정.

세부: `risk.yaml`, `yolo.yaml`, `aihub_to_yolo.py` 주석.

## 7. 통계 (Sample 330장 기준)

### 클래스 인스턴스 분포

| HelpBee 클래스 | 71667 출처 | 인스턴스 | 비율 |
|---|---|---|---|
| bee_normal (0) | 유충_정상 + 성충_정상 | 3,718 | **88%** |
| bee_with_varroa (1) | 유충_응애 + 성충_응애 | 100 | **2.4%** |
| bee_other_disease (2) | 부저병 + 석고병 + 날개불구 | 392 | 9.3% |

→ 응애 클래스 심각한 imbalance. **Copy-Paste augmentation (yolo.yaml `copy_paste: 0.3`)
필수**. 5,000장 확장 시에도 응애 ~250개 수준 → oversampling 검토 가능.

### 메타 분포

- **colony.id**: 9개 unique. "001"이 249/330 = **75%** 점유 (heavy imbalance)
- **device**: 소비판촬영기(57%) / 플레이트촬영기(39%) / 소문촬영기(3%)
- **datetime**: 2023-08~ 시간대 다양

### bbox 크기 분포 (전체)

- ratio < 1% (작은 객체): 35개
- 1~5% (소): 367개 + ...
- 5~30% (중, 한 마리 벌): 1,703개 + 161개(질병)
- ≥ 30% (대, 프레임 일부): 1,639개 + 205개(질병)

## 8. 분할 전략 — data leakage 방지

**같은 colony 사진이 train/val 양쪽에 있으면 mAP가 거짓말한다.**

`split_strategy.py` 가 메타 + 분포 자동 감지:

| 전략 | 적용 조건 | 이 데이터셋에 적합? |
|---|---|---|
| **per_colony_time_block** ★ | farm_id + captured_at ≥ 80% + colony imbalance | ✅ default 권장 |
| farm_group | farm_id ≥ 80% (균형) | ⚠️ 001=75% 라 비율 깨짐 |
| device_holdout | capture_device ≥ 80% | 보조 (device generalization 평가용) |
| sequence_block | captured_at 만 풍부 | fallback |
| random | 메타 없음 | ⚠️ leakage 위험 — 절대 first choice X |

### per_colony_time_block (권장)

각 colony 내에서 시간 순으로 마지막 val_ratio% 를 val.

이점:
- colony 단위 누수 차단 (모든 colony 가 양쪽에 시간 분리되어 들어감)
- 한 농가가 75% 차지해도 그 농가의 80%는 train, 20%는 val 로 균형 유지
- 시간 분리로 같은 세션 누수 방지

가정: 같은 colony 의 다른 시간대 사진이 분포 변화를 충분히 반영. 같은 세션 내
연속 프레임은 여전히 시간이 매우 가깝게 잘려 누수 위험 잔존 — 베타에서 외부 데이터로 보강.

## 9. Golden Holdout 셋

`golden_holdout.py` 로 학습/aug에서 영구 제외할 평가 셋 추출.

- 응애 100장 + 정상 200장 = **300장**
- 다양성 강제: 농가 ≥ 3, 촬영 기기 ≥ 2
- 71667 외부 데이터(베타 양봉가 사진) 확보되면 그것으로 교체 권장 — 같은 데이터셋
  내부 분리는 분포 동질이라 일반화 평가가 약함.

## 10. 위험도 측정 정의

71667 라벨 특성 때문에 표준 VMIR 가 아닌 **infestation_rate** 사용:

```
infestation_rate (%) = (bee_with_varroa 인스턴스 수) / (모든 벌 인스턴스 수) × 100
                       분모 = bee_normal + bee_with_varroa + bee_other_disease
```

임계값 (`risk.yaml`):
- < 3% safe / 3~10% watch / ≥ 10% danger

표준 VMIR(mites per 100 bees, Honey Bee Health Coalition) 임계값을 차용했으나
**측정 정의가 다름** — 베타 양봉가 사진에 sugar-roll/alcohol-wash 실측 VMIR 라벨링 후
회귀 보정 필요.

## 11. 알려진 제약 / 미해결

- **응애 자체 bbox 없음** — 응애 객체 카운팅 task 는 이 데이터셋만으로 불가
- **colony 001 dominance (75%)** — 일반화 평가 약화. 외부 데이터로 보강 필수
- **응애 인스턴스 부족 (2.4%)** — Copy-Paste + oversampling 필수
- **bbox 크기 거의 모두 medium~large** — 작은 응애 탐지 모델로는 학습 시그널 부족
- **state / symptoms 메타 활용 안 함** — v0.2.0+ 부가 진단 정보로 활용 가능
- **벌통 외부 환경 (in/out temp/humidity) 안 씀** — multimodal 학습 후보 (v0.3.0+)
- **사용 제약 (내국인만)** — 국제 협업 시 별도 데이터 필요
